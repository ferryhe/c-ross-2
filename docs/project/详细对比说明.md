# 详细对比说明：面向法规 Markdown 的 Agent Chatbot、Agent RAG 与 Skill 化回答方案

更新时间：2026-04-03

## 1. 文档目标

这份说明不是单纯做“市面产品罗列”，而是要回答下面几个更实际的问题：

- 现在主流的 agent chatbot / agent assistant，分别擅长什么。
- 它们和本项目现有的 `agentic RAG` 相比，哪些能力值得借鉴，哪些反而不适合直接照搬。
- 对于“法规标题明确、正文长、附件多、公式多、通知和规则之间有覆盖关系”的 Markdown 知识库，怎样把 `title 搜索 + summary 搜索 + RAG` 结合起来。
- 如果未来不是只做聊天机器人，而是要适配类似 OpenClaw 这类“可挂 skill、可调工具、可自主拆解任务”的 AI agent，应该怎样定义 skill，才能让 agent 真正回答出用户想要的内容。

本文把“类似 OpenClaw 的 AI agent”理解为一种更通用的运行方式：

- Agent 不只会聊天。
- Agent 可以调用多个检索/数据工具。
- Agent 可以根据 skill 按步骤做任务拆解、证据收集、回答组织和自检。

如果这个理解和你后续接入的具体框架略有差异，本文的方法仍然成立，因为核心不是某个产品，而是“法规型知识库的可验证回答机制”。

## 2. 先给结论

### 2.1 一句话结论

对于你这个仓库里的法规、通知、附件、公式型 Markdown，最优方案不是“单一向量检索 + 聊天回答”，而是：

`标题检索 -> 摘要检索 -> 结构化证据检索 -> 交叉引用扩展 -> 带约束的回答模板 -> skill 化流程控制`

### 2.2 更具体的结论

1. 你现在的项目已经比很多“普通聊天机器人接知识库”的方案更接近真正可用的法规 agent。
2. 现有实现的长处在于：文档标准化、公式修复、文档级与 section 级双索引、针对“规则第X号/附件X”这类问题的直达匹配、以及带引用回答。
3. 现阶段最明显的短板不是模型不够强，而是“结构化中间层还不够厚”。
4. 对法规问答，最缺的不是更多 prompt，而是：
   - 更强的标题/别名/编号命中层
   - 更可靠的文档摘要层
   - 更细的条文/公式/表格/阈值/例外层
   - 更清楚的规则与通知、正文与附件之间的关系层
5. 如果未来要做 skill 化 agent，应该把“问答”拆成多个能力，而不是只暴露一个 `ask(question)`。

## 3. 当前项目的实际基础

基于当前仓库，我看到的现状是：

- 知识库目录 `Knowledge_Base_MarkDown/` 已经按 `rules / notices / attachments` 分层。
- 当前库内约有 `20` 份规则、`7` 份通知、`5` 份附件 Markdown。
- 文档前置处理已经包含 front matter、分类、来源、发布时间等元数据。
- 数学公式已经做过专门清洗，包括 HTML entity 解码、中文下标修复、`cases` 结构重写、部分行内公式普通化。
- RAG 侧不是纯标准模式，而是已经有轻量 `agentic RAG`：
  - query planning
  - reflection
  - rerank
  - hybrid document/section retrieval
  - filename hint boosting
- 前端已经支持：
  - KaTeX 公式渲染
  - 数字引用跳转
  - section 级来源展示

这意味着你并不是从“零到一”做法规 agent，而是已经完成了第一段最难但最容易被低估的工作：把混乱监管材料整理成可检索的规范 Markdown。

### 3.1 当前方案的真正优势

- 域内非常窄，语义漂移比公网搜索小得多。
- 文件名本身就是强信号，例如“规则第2号”“附件4”“通知”。
- 文档、章节、公式、表格都在同一知识域里，天然适合交叉引用。
- 当前回答系统强调引用，不是纯生成。
- 公式渲染和公式清洗已经存在，这对法规类问答非常关键。

### 3.2 当前方案的主要缺口

- 还没有“文档摘要索引”这一层。
- 还没有“条文 / 公式 / 表格 / 阈值 / 例外 / 报送要求”的结构化标签层。
- 还没有“规则被通知修正、附件为正文补充、公式出自哪一条”的关系图层。
- 当前检索策略已经有 `document / section / hybrid` 路由，但还不够细，不足以覆盖法规场景的核心问题类型。
- 现有回答更像“研究型 RAG 回答”，还不是“法规型 agent 回答”。

## 4. 市面主流 agent 产品，对你这个场景的参考价值

以下不是“谁更强”的排行榜，而是“谁更值得借鉴到你的法规知识库方案里”。

### 4.1 总览表

| 产品/形态 | 目前公开能力重点 | 对法规 Markdown 场景的优点 | 对法规 Markdown 场景的局限 | 对本项目最值得借鉴的点 |
|---|---|---|---|---|
| ChatGPT Deep Research + Apps/Agent Mode | 多步研究、连接外部应用、可限制站点、可导出报告 | 研究流程透明、可看计划、可限制来源、引用较强 | 更偏广域研究，不天然理解“第几号规则/哪条/哪张表/哪个公式” | “计划可编辑 + 来源可限制 + 报告可复查” |
| Claude Research + Connectors + Skills | MCP 连接器、Research、Skills、工具自动调用 | 很适合做“程序化工作流 + 专业技能封装” | 默认不是法规专用，仍需要结构化知识层支撑 | “Skill 与 MCP 分层”最值得学 |
| Gemini Deep Research + Gems | 网页研究、计划可视化、个性化 Gems | 适合做角色化问答和多步调研 | 更偏 web research，对本地法规结构粒度没有天然优势 | “角色化配置 + 研究过程可见” |
| NotebookLM | 严格依托用户给定来源、支持 Markdown、支持自定义回答风格 | 对文档问答、可信引用、长文总结非常合适 | 更像 source-grounded notebook，不是强工作流 agent | “基于可信来源回答”的产品心智很适合你 |
| Microsoft 365 Copilot + Agents | 租户知识、Graph grounding、Agent Builder、网站/连接器作为知识 | 企业知识治理、权限控制、知识与工具分离做得强 | 更偏企业办公流，法规公式解释并非其默认强项 | “知识源/工具/权限分层治理”值得借鉴 |

### 4.2 ChatGPT Deep Research

适合参考的原因：

- 它把“研究任务”和“即时聊天”分开了。
- 它允许用户在研究开始前查看并调整研究计划。
- 它支持连接文档库和受认证的数据源。
- 它允许限制 research 只搜索指定网站，或者优先搜索指定网站。
- 输出会附带引用、来源列表和活动历史。

这对你的启发非常直接：

- 法规问答不应该一上来就“直接回答”。
- 先锁定检索范围，再做 synthesis，比直接大模型自由发挥更可靠。
- 对监管场景，限制来源范围非常重要，尤其当未来接入公网或外部库时。

但 ChatGPT 的默认能力仍然更偏“广域研究 agent”，而不是“法规条文定位 agent”。如果把它直接照搬到你这里，很容易出现：

- 结论有道理，但不一定落在正确规则条文上。
- 能总结，但未必优先引用最权威、最具体的条款。
- 能做 research，但不天然知道“通知可能修正规则正文”。

### 4.3 Claude Research + Connectors + Skills

Claude 这条路线对你最有参考价值，因为它把三个概念拆得很清楚：

- Connectors / MCP：负责连数据和工具。
- Research：负责多步调查和证据收集。
- Skills：负责“怎么做这件事”的程序化方法。

这和你的目标高度一致，因为你真正需要的不是更会聊天，而是：

- 让 agent 知道什么时候先找标题。
- 什么时候先看摘要。
- 什么时候必须找公式块、变量定义和相关附件。
- 什么时候必须补查通知与规则之间的覆盖关系。

也就是说，你最该借鉴 Claude 的不是某个模型，而是“把工作流知识外置成 skill”这件事。

### 4.4 Gemini Deep Research + Gems

Gemini 的 Deep Research 和 Gems 组合，适合参考的点在于：

- Deep Research 更强调计划、搜索、推理、分析、报告是一整条链。
- Gems 则更像“角色化的轻 skill”。

对你的场景来说，它给出的启发是：

- 可以把“监管规则解释员”“公式说明员”“报送要求核对员”“规则差异对比员”做成不同 skill/persona。
- 同一份知识库，不同问题类型应有不同回答套路，而不是统一 prompt。

但它也提醒了一个边界：

- 只有 persona 没有结构化证据层，最终还是容易流于“会说但不够可验证”。

### 4.5 NotebookLM

如果只看“法规 Markdown 问答”，NotebookLM 其实比很多通用 chatbot 更值得参考。

因为它的核心心智不是“我懂很多”，而是：

- 你给我哪些来源，我就在这些来源里回答。
- 我更像一个依托可信材料的研究助手。

这和法规场景天然贴近。对这类问题，用户真正想要的不是“世界知识”，而是：

- 这条规定在哪。
- 原文是什么意思。
- 适用范围是什么。
- 有没有例外、通知、附件、公式、阈值、报送要求。

所以从产品方法论上看，你的项目更应该向“NotebookLM + structured RAG + skill 化 agent”靠，而不是向“万能聊天机器人”靠。

### 4.6 Microsoft 365 Copilot + Agents

微软路线的价值不在“回答更聪明”，而在“知识、工具、权限、治理是分层的”。

它适合借鉴的地方有：

- 知识源与工具分开定义。
- 不同 agent 可以加载不同知识源和工具。
- 网站、租户数据、Graph connector 都可以是 agent knowledge。
- 安全与治理被视为 agent 设计的一部分，而不是后补。

这对法规知识库非常重要，因为法规回答不是纯 QA，它涉及：

- 权威性
- 来源边界
- 版本关系
- 通知是否覆盖原条文
- 是否允许 agent 执行外部动作

## 5. 市场对比后的核心判断

### 5.1 市面产品普遍更强的地方

- 通用 web research 能力更强。
- 多工具调度更成熟。
- 用户体验更完整，尤其是研究过程可视化、可中断、可导出。
- 企业权限、应用连接器和跨系统上下文更成熟。

### 5.2 你这个项目天然更强的地方

- 语料边界清楚。
- 中文监管术语一致。
- 文档标题本身高度结构化。
- 可直接做本地精细索引，不必依赖公网。
- 可以把条文、通知、附件、公式之间的关系做得比通用产品更细。

### 5.3 真正的机会点

真正值得做的不是把本项目做成“更像 ChatGPT/Claude 的聊天页”，而是把它做成：

`对法规型 Markdown 更懂结构、更会定位、更会引用、更会解释公式、更会处理例外和覆盖关系的专业 agent`

## 6. 为什么法规知识库不能只做普通 RAG

普通 RAG 通常假设：

- 文本块是独立的。
- 语义相似就够了。
- 只要取到几个相关 chunk，大模型就能回答。

但法规场景不满足这个假设，因为法规文本有几个特别强的结构特征：

### 6.1 标题和编号本身就是一级检索信号

例如：

- `保险公司偿付能力监管规则第2号：最低资本`
- `附件4：保险公司压力测试必测压力情景和必测因素`
- `关于优化保险公司偿付能力监管标准的通知`

这些标题不是普通文档标题，而是用户提问时最常被直接引用的“正式定位入口”。

所以如果只用 dense retrieval，而没有强标题检索层，就会浪费最有用的信号。

### 6.2 摘要需求和条文定位需求是两种完全不同的任务

用户问：

- “规则第2号主要讲什么”

和用户问：

- “规则第2号第十五条的计算公式是什么”

它们不是同一类检索任务。

前者需要：

- 锁定文档
- 阅读全篇
- 形成结构化摘要

后者需要：

- 锁定具体条文
- 找到公式块
- 回收变量解释
- 找到上下文条件和附件关系

所以必须把“summary 检索”和“evidence 检索”分层。

### 6.3 公式不是普通段落

对于法规中的公式，用户通常真正想问的是以下几种之一：

- 这个公式在哪里。
- 这个公式每个变量是什么意思。
- 这个公式适用于什么业务类型。
- 上限/下限/阈值/相关系数是多少。
- 如果给定参数，怎么计算。

如果把公式只当成普通文本块处理，会有三个问题：

- 检索经常命中不到。
- 命中后也不容易把变量解释一起带出来。
- 回答容易只复述公式，不解释其业务意义和适用条件。

### 6.4 通知、规则、附件之间有覆盖关系

法规场景里，最危险的错误不是“没答出来”，而是：

- 答了一个看似正确但版本不完整的答案。

例如：

- 规则正文给出一般原则。
- 后续通知调整参数或过渡期。
- 附件给出表格、曲线、名单、模板。

这意味着检索层必须能回答：

- 原始规定是什么。
- 后续通知有没有调整。
- 具体数字或模板是不是在附件里。

## 7. 建议的新架构：标题搜索 + Summary 搜索 + 结构化 RAG + 关系图

我建议把未来方案拆成四层数据、两层流程。

### 7.1 四层数据

#### 第一层：目录与标题层

目标：先解决“用户到底在问哪份文档”。

建议字段：

- `doc_id`
- `title`
- `aliases`
- `category`
- `rule_no`
- `attachment_no`
- `notice_type`
- `authority`
- `publish_date`
- `effective_date`
- `priority_rank`
- `path`

这个层要支持：

- 精确标题命中
- 编号命中
- 别名命中
- 中文简写命中
- 文件路径权重加成

#### 第二层：摘要层

目标：先快速判断“这几份文档里谁更相关、谁更适合做 overview”。

建议每份文档额外生成两个摘要：

- `summary_short`
  - 80 到 150 字
  - 适合路由与粗召回
- `summary_structured`
  - 200 到 500 字
  - 按“主题 / 适用对象 / 核心要求 / 重要参数 / 相关附件 / 后续通知”组织

这层主要用于：

- `主要内容`
- `概览`
- `适用范围`
- `这份通知调整了什么`
- `规则第2号和第4号分别讲什么`

#### 第三层：结构化证据层

目标：解决真正回答时需要的“硬证据”。

建议把每份 Markdown 按更细粒度拆成以下类型：

- `chapter`
- `article`
- `clause`
- `definition`
- `formula`
- `table`
- `threshold`
- `obligation`
- `exception`
- `reporting_requirement`
- `timeline`

每条证据建议带这些字段：

- `doc_id`
- `path`
- `heading_path`
- `article_no`
- `clause_no`
- `content_type`
- `text`
- `formula_latex`
- `variables`
- `keywords`
- `source_kind`

#### 第四层：关系图层

目标：让 agent 不只“检索相似文本”，还能“顺着法规关系走”。

建议维护这些关系：

- `rule -> attachment`
- `rule -> notice`
- `notice -> adjusted_rule`
- `formula -> variable_definition`
- `article -> referenced_attachment`
- `doc -> same_topic_docs`
- `rule -> superseded_by`

### 7.2 两层流程

#### 流程 A：定位流程

适用于：

- “规则第2号是什么”
- “附件4在哪”
- “关于优化监管标准的通知”

流程：

1. 编号/标题/别名优先匹配
2. 摘要层二次判定
3. 返回候选文档及理由

#### 流程 B：回答流程

适用于：

- 摘要
- 解释
- 对比
- 公式说明
- 报送要求
- 适用性判断

流程：

1. 先做定位
2. 再做摘要或结构化证据检索
3. 必要时扩展到通知、附件、相关条文
4. 最后按问题类型套回答模板

## 8. 问题类型与检索策略，应该一一对应

这是整个方案里最关键的一点。

| 问题类型 | 典型问法 | 首选检索层 | 补充检索层 | 推荐回答结构 |
|---|---|---|---|---|
| 定位型 | “规则第2号在哪” | 标题层 | 摘要层 | 文档定位 + 文件类别 + 一句话说明 |
| 摘要型 | “规则第2号主要内容” | 标题层 + 摘要层 | 文档层 | 结论 + 主题拆分 + 相关附件/通知 |
| 条文型 | “第十五条说了什么” | 标题层 + article 层 | 文档层 | 条文原意 + 适用范围 + 相关上下文 |
| 公式型 | “计算公式是什么” | 标题层 + formula 层 | article 层 + attachment 层 | 公式 + 变量定义 + 条件 + 如需计算所缺输入 |
| 对比型 | “规则2和4区别” | 标题层 + 摘要层 | article 层 | 相同点/不同点/适用对象/关联关系 |
| 合规型 | “要不要报送/披露” | 摘要层 + obligation 层 | exception + timeline + notice | 结论 + 依据 + 条件 + 例外 + 时间要求 |
| 版本型 | “通知调整了什么” | notice 层 + summary 层 | related rule/article | 调整对象 + 调整内容 + 生效影响 |

## 9. 对公式密集型 Markdown，必须单独设计

### 9.1 入库阶段要做的不是“保留公式”，而是“解释公式”

建议新增一个 `formula card` 概念。

每个公式卡片至少包含：

- `formula_id`
- `doc_id`
- `article_no`
- `heading_path`
- `latex`
- `plain_name`
- `variables`
- `variable_explanations`
- `applicable_scope`
- `related_tables`
- `related_attachments`

例如对 `MC*`、`LA`、`β`、`ρ` 这类符号，不应该只存公式正文，还应该补：

- 变量是什么
- 变量在哪段定义
- 参数取值是否固定
- 是否存在上下限
- 是否依赖附件表格或曲线

### 9.2 检索阶段要允许“变量名检索”

很多用户不会问“第十九条公式是什么”，而会问：

- `LA 上限怎么定义`
- `β 怎么取值`
- `ρ 是多少`

所以公式层要支持：

- 符号检索
- 中文变量解释检索
- 英文/LaTeX 变量检索

### 9.3 回答阶段不要只贴公式

对公式问题，推荐固定回答结构：

1. 先给结论
2. 给出公式
3. 解释每个变量
4. 说明适用条件
5. 指出是否还要查附件/通知
6. 如果用户要“算”，明确还缺哪些输入

也就是说，用户问“公式是什么”，真正想要的通常是“这个公式怎么用”，不是只要一段 LaTeX。

## 10. 更适合 AI agent 的回答方法，不应只靠 prompt

### 10.1 回答方法应该是“结构 + 证据 + 模板”的组合

推荐回答框架：

- `直接结论`
- `依据来源`
- `适用范围`
- `条件/例外`
- `相关附件/通知`
- `如果涉及公式：变量定义与计算说明`
- `如果证据不足：明确缺什么`

### 10.2 对法规问题，回答必须学会说“不够信息”

好的 agent 不是“什么都答”，而是：

- 能明确告诉用户当前证据只支持到哪一步。
- 能明确下一步应该查哪一份 Markdown、哪一条、哪一个附件。

这在法规场景里非常重要，因为“不确定但胡说”比“没答出来”风险大得多。

### 10.3 对比问题必须显式分栏

用户问对比类问题时，不要把两份法规混在一段话里。

建议输出结构固定为：

- 比较对象
- 共同点
- 差异点
- 适用对象差异
- 是否存在覆盖/补充关系
- 依据条文

### 10.4 合规判断问题必须区分“规则要求”和“业务事实”

很多用户问的是：

- “这家公司要不要报送”
- “这个情形算不算”

对这类问题，agent 应当分开写：

- 规则层结论：法规怎么规定
- 事实层假设：当前已知事实是什么
- 差距层：还缺哪些事实，不能直接下结论

## 11. 适配类似 OpenClaw 的 AI Agent：不要只暴露一个问答接口

如果未来真要接 skill 型 agent，我建议把能力拆成多个工具，而不是只有一个黑盒问答。

### 11.1 建议暴露的工具

- `search_titles(query)`
  - 根据标题、编号、别名找候选文档
- `search_summaries(query, doc_ids=None)`
  - 根据结构化摘要找相关文档
- `search_articles(query, doc_ids=None)`
  - 查具体条文、章节、定义
- `search_formulas(query, doc_ids=None)`
  - 查公式、变量、相关条文
- `search_tables(query, doc_ids=None)`
  - 查表格、阈值、名单、曲线
- `expand_relations(doc_or_section_ids)`
  - 查相关通知、附件、被调整关系
- `compose_regulatory_answer(question, evidence_pack, answer_type)`
  - 只负责按模板组织答案，不再负责盲检索

这样设计的好处是：

- skill 可以明确控制流程。
- agent 能解释自己为什么这样查。
- 方便做评估和调试。
- 未来替换模型时，流程资产仍然保留。

### 11.2 skill 应该教 agent 什么

skill 最重要的不是“告诉 agent 监管知识本身”，而是“告诉它怎么处理监管知识”。

建议 skill 至少包含以下规则：

- 当问题含有“规则第X号/附件X/通知”时，必须先跑标题定位，不得直接语义检索。
- 当问题是“主要内容/概览/总结”时，必须优先读 summary，而不是拼接零散 chunks。
- 当问题含有“公式/计算/系数/阈值/上限/下限/取值”时，必须优先走 formula/table 检索。
- 当问题涉及“调整/延长/优化/实施过渡期”等词时，必须补查 notice 层。
- 当回答带有结论性语句时，必须给出处。
- 当证据只支持部分结论时，必须显式标出“不足部分”。

### 11.3 一份可直接改写的 Skill 草案

```md
---
name: regulatory-markdown-answering
description: 用于处理法规、通知、附件、公式密集型 Markdown 知识库的问题回答 skill。
when_to_use: 当用户询问规则内容、适用范围、公式、参数、阈值、报送要求、通知调整、附件关系、规则对比时启用。
---

# Goal

给出可验证、可定位、对业务人员有用的法规回答，而不是泛泛总结。

# Mandatory workflow

1. 先判断问题类型：定位 / 摘要 / 条文 / 公式 / 对比 / 合规 / 版本。
2. 如果问题包含规则号、附件号、通知名或正式标题，先调用标题检索。
3. 如果问题是摘要类，先调用 summary 检索，再决定是否补拉全文或 sections。
4. 如果问题涉及公式、变量、系数、阈值、取值、上下限，优先调用公式和表格检索。
5. 如果问题涉及调整、修订、延长、实施、过渡期，补查 notices 和关系图。
6. 组织答案时必须区分：
   - 直接结论
   - 依据条文/文档
   - 适用范围
   - 条件与例外
   - 仍不确定的部分
7. 没有证据支持的内容不得推断为法规结论。

# Answer rules

- 不要只贴公式，要解释变量和适用条件。
- 不要只给总结，要说明来自哪份规则/通知/附件。
- 对比类问题必须分栏说明相同点和不同点。
- 合规判断类问题必须区分“法规要求”和“当前事实假设”。
- 如果证据不足，明确告诉用户下一步应查哪份 Markdown、哪一条或哪个附件。
```

## 12. 针对你这个仓库，建议新增哪些中间产物

如果希望回答质量明显提升，我建议在 Markdown 之外，再新增一些 sidecar 数据。

### 12.1 `doc_summaries.jsonl`

每份文档一条，包含：

- `doc_id`
- `title`
- `category`
- `summary_short`
- `summary_structured`
- `keywords`
- `related_docs`

### 12.2 `doc_title_aliases.jsonl`

包含：

- 正式标题
- 简称
- 编号形式
- 用户口语形式

例如：

- `规则第2号`
- `最低资本规则`
- `偿二代规则2`

### 12.3 `sections_structured.jsonl`

每条 section 带：

- `article_no`
- `content_type`
- `heading_path`
- `text`
- `variables`
- `thresholds`
- `obligations`

### 12.4 `relations_graph.json`

用于记录：

- 正文与附件
- 规则与通知
- 条文与公式
- 公式与变量定义
- 文档之间的补充/修订/关联关系

## 13. 实施路线图

### 阶段 1：补 metadata，不改前台交互

目标：

- 新增标题别名层
- 新增摘要层
- 新增 doc-level structured summary

价值：

- “主要内容/概览/对比”类问题会立刻明显变好。

### 阶段 2：补结构化 section 抽取

目标：

- 把 `article / formula / table / threshold / obligation / exception` 单独打标

价值：

- 公式、参数、阈值、报送要求类问题会明显变好。

### 阶段 3：补关系图

目标：

- 处理通知调整、附件关联、规则补充

价值：

- 降低“答了旧版本结论”的风险。

### 阶段 4：把问答接口拆成 agent tools

目标：

- 让 skill 可控地调用 `title -> summary -> evidence -> relation -> answer`

价值：

- 更适合接入 OpenClaw 这类 skill/agent 框架。

### 阶段 5：做专项评测集

至少覆盖：

- 规则概览
- 精确条文定位
- 公式解释
- 参数/阈值提取
- 规则对比
- 通知覆盖
- 报送要求
- 证据不足场景

## 14. 最终建议：把它做成“法规型回答引擎”，而不是“又一个聊天机器人”

如果只想做一个能聊天的界面，你现在其实已经差不多了。

但如果你的目标是：

- 更稳定
- 更可验证
- 更适配法规
- 更适配 skill/agent
- 更擅长处理公式和附件

那么下一步最值得投入的不是“换更大的模型”，而是：

1. 强化标题层
2. 新增摘要层
3. 新增公式/表格/条文结构化层
4. 补齐规则-通知-附件关系层
5. 用 skill 把回答流程固定下来

到那一步，这个项目的定位就不再是“RAG 聊天机器人”，而会更接近：

`面向监管规则知识库的、可验证、可扩展、适合 AI agent 调用的专业回答引擎`
