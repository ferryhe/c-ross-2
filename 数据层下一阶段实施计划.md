# 数据层下一阶段实施计划

目标：继续把当前仓库收口为

`MD -> chatbot-ready / AI-agent-ready 数据层 + 可验证 RAG 层`

并保持现有独立 chatbot 作为验证 consumer。

## 对齐的 skill 约束

本计划按以下项目内 skill 思路推进：

- `skills/regulatory-ready-data-builder/SKILL.md`
- `skills/regulatory-markdown-answering/SKILL.md`

两者分别约束：

- Markdown 如何被稳定加工成 ready-data 产物
- AI agent 如何基于这些产物做标题定位、摘要缩圈、证据检索和可验证回答

## 当前基线

当前已经具备：

- Markdown 主库：`Knowledge_Base_MarkDown/`
- hybrid document/section 向量检索
- 专业引擎骨架：title search / summary search / query planning
- ready-data builder 第一版
- ready-data 产物第一版：
  - `doc_catalog.jsonl`
  - `title_aliases.jsonl`
  - `doc_summaries.jsonl`
  - `sections_structured.jsonl`
  - `formula_cards.jsonl`
  - `relations_graph.json`
  - `ready_data_manifest.json`

## 下一阶段目标

下一阶段不再把重点放在“多做几个接口”，而是放在下面四件事：

1. 稳定数据契约
2. 强化结构化产物
3. 让 chatbot / AI agent 都能消费同一套数据层
4. 建立真实问答验证闭环

## Phase 1：稳定数据契约

目标：

- 把 ready-data 变成这个 repo 最稳定的主产物
- 让 consumer 优先依赖 ready-data，而不是每次回到原始 Markdown 现算

本阶段动作：

- 冻结 `doc_catalog / title_aliases / doc_summaries / sections_structured / formula_cards / relations_graph` 的基础字段
- 统一 `doc_id`、`path`、`section_id`、`formula_id` 的稳定规则
- 规定 manifest 中必须包含产物计数、构建时间、source root
- 为 ready-data 增加最小可回归测试

建议提交拆分：

1. schema 文档
2. builder 修正
3. ready-data 产物重建

## Phase 2：增强结构化产物

目标：

- 让数据层更适合法规、通知、附件、公式、阈值类问题

本阶段动作：

- 在 `sections_structured.jsonl` 中继续强化：
  - `content_type`
  - `article_no`
  - `clause_no`
  - `mentions_rules`
  - `mentions_attachments`
- 为公式密集文档补充：
  - `formula_cards.jsonl` 的变量别名
  - 公式邻近条文引用
  - 公式适用范围说明
- 为通知和规则关系补充：
  - `relation = adjusts_rule`
  - `relation = extends_transition`
  - `relation = clarifies_formula`
- 为附件补充：
  - `attachment_subject`
  - `applies_to_rule`

建议提交拆分：

1. section / formula 提取增强
2. relations graph 增强
3. ready-data 产物重建

## Phase 3：统一 consumer 接入

目标：

- 当前 chatbot 保留
- 新专业引擎继续增强
- 两者共用同一套 ready-data 和向量层

本阶段动作：

- 当前 chatbot：
  - 保留 `/api/chat`
  - 继续作为“结果可行性验证器”
- 专业引擎：
  - 优先使用 `doc_catalog` 做标题和编号定位
  - 使用 `doc_summaries` 做摘要缩圈
  - 使用 `sections_structured` 和 `formula_cards` 做细证据召回
  - 使用 `relations_graph` 做规则/通知/附件扩展
- 对编号明确的问题启用严格 scope，避免召回发散

建议提交拆分：

1. professional engine 消费 ready-data
2. chatbot 兼容和回归测试
3. API 文档更新

## Phase 4：真实问答验证闭环

目标：

- 不只验证“能运行”
- 验证“回答是否非常接近监管文本真实含义”

本阶段动作：

- 固定一组关键问题：
  - 规则数量类
  - 单规则概览类
  - 最低资本构成类
  - 公式解释类
  - 附件定位类
  - 通知/规则关系类
- 固定评测模型：
  - `gpt-5.4-mini`
- 每轮记录：
  - 问题
  - 回答
  - 命中的路径
  - 人工结论：正确 / 基本正确 / 部分偏题 / 错误
  - 是否需要改 schema / retrieval / prompt

建议提交拆分：

1. 评测脚本
2. 真实问答记录
3. 根据评测修 retrieval 或 prompt

## 本轮之后最值得继续做的顺序

1. 把 ready-data 构建结果纳入版本管理
2. 增加真实问答评测脚本
3. 用 `gpt-5.4-mini` 跑关键问题并记录
4. 根据评测结果继续收紧 exact rule / attachment scope
5. 再补通知覆盖关系和公式变量说明

## 验收标准

达到下面四点，就说明这一阶段完成得比较扎实：

- ready-data 可以独立重建
- 专业引擎优先消费 ready-data
- 编号明确的问题不会明显召回跑偏
- 至少一组 `gpt-5.4-mini` 真实问答记录能够证明关键问题回答基本准确
